# 医学病例合规审核 Agent (Medical Audit Agent)

本项目是一个自动化医学审核 Agent，用于辅助医学经理进行合规质检。输入为临床医生提交的病例 PPT，通过大模型与自定义的药品规则库进行比对，最终输出结构化的合规审核意见（HTML），支持 PDF 导出。

---

## 1. 核心系统架构

采用四阶段线性 Pipeline，支持并行处理多个 PPT 文件：

```
[输入: data/input_cases/*.pptx (批量)]
       │
       ▼
┌────────────────────────────────────────────────────────┐
│ 1. 文本提取 (python-pptx)                               │
│    - 提取 PPT 文本框、表格数据                           │
└────────────────────────────────────────────────────────┘
       │
       ▼
┌────────────────────────────────────────────────────────┐
│ 2. 图片提取 (python-pptx)                               │
│    - 提取嵌入图片，保存至 output_reports/images/         │
│    - 自动跳过不支持的图片格式（MPO等）                   │
│    - 可选：调用视觉模型识别图片内容（需配置多模态模型）    │
└────────────────────────────────────────────────────────┘
       │
       ▼
┌────────────────────────────────────────────────────────┐
│ 3. 结构化 (LLM)                                         │
│    - 文本+图片描述 → 结构化患者 JSON                     │
│    - 约束药品名和适应症必须匹配规则库候选值               │
│    - 自动清理 LLM 输出中的格式问题（JSON 容错+重试）      │
└────────────────────────────────────────────────────────┘
       │
       ▼
┌────────────────────────────────────────────────────────┐
│ 4. 合规审核 (Rule Matcher + LLM)                        │
│    - 基础质控检查（必选）：                               │
│      · 文本规范性检查（错别字、语法）                     │
│      · 通用名规范检查（药品通用名 vs 商品名）             │
│      · 医学逻辑审核（药物与适应症匹配性）                 │
│      · 单位规范检查（剂量、频率、检验指标单位）           │
│    - 合规专项审核（按规则库）：                           │
│      · 适应症合规、隐私保护、Storyline故事线              │
│      · 安全性风险、科学幻灯、信息完整性等                 │
└────────────────────────────────────────────────────────┘
       │
       ▼
┌────────────────────────────────────────────────────────┐
│ 5. 报告生成 (Jinja2)                                    │
│    - 渲染为专业风格的 HTML 审核报告                      │
│    - 分组展示：基础质控检查 + 合规专项审核                │
│    - 支持 PDF 导出（浏览器打印功能）                     │
│    - 含红绿灯状态徽章、隐私风险告警、统计摘要            │
└────────────────────────────────────────────────────────┘
       │
       ▼
[输出: data/output_reports/audit_report_combined.html]
```

## 4. API 配置

复制 `.env.example` 为 `.env`，填写你的 API Key：

```env
OPENAI_BASE_URL=https://token-plan-cn.xiaomimimo.com/v1
OPENAI_API_KEY=your_api_key_here
OPENAI_MODEL_ID=mimo-v2.5-pro
OPENAI_MULTIMODAL_MODEL=          # 视觉模型，为空则跳过图片识别
```

**视觉模型说明**：当前 Mimo API 不支持图片输入，`OPENAI_MULTIMODAL_MODEL` 留空即可。后续如有支持 vision 的模型，设置该变量即可启用图片识别能力。

## 6. 核心运行脚本 (main.py)

### 运行方式

```bash
python main.py
```

### 功能特性

| 特性 | 说明 |
|------|------|
| **并行处理** | 使用 ThreadPoolExecutor 并行处理多个 PPT（最多 3 线程） |
| **批量处理** | 自动扫描 `data/input_cases/` 下所有 `.pptx` 文件 |
| **图片提取** | 提取 PPT 中嵌入的图片，自动跳过不支持的格式 |
| **基础质控检查** | 4 项固定检查：文本规范性、通用名、医学逻辑、单位规范 |
| **合规专项审核** | 基于规则库的多维度审核（适应症、隐私、安全性等） |
| **JSON 容错** | 自动清理 LLM 输出格式问题，失败时自动重试（最多 2 次） |
| **日志系统** | 树形结构日志，线程锁确保并行时日志不交错 |
| **HTML 报告** | 专业风格，分组展示，支持 PDF 导出 |

### 处理流程

```
▶ 开始处理: 文件名
  ├─ [1/4] 解析PPT文本...
  ├─ [2/4] 提取图片...
  │  提取图片 N 张
  ├─ [3/4] LLM结构化患者数据...
  │  用药: 药品名 | 适应症: 适应症名
  └─ [4/4] 合规审核...
     结果: ✓N Pass | ⚠N Warning | ✗N Fail
```

### 审核维度说明

**基础质控检查（必选，每份报告都有）：**

| 检查项 | 说明 |
|--------|------|
| 文本规范性检查 | 检查错别字、语法错误、标点符号误用、医学术语拼写错误 |
| 通用名规范检查 | 检查药品是否使用通用名（INN），而非商品名/品牌名 |
| 医学逻辑审核 | 检查药物与适应症匹配性、剂量合理性、疗程合规性、联合用药风险 |
| 单位规范检查 | 检查剂量单位、给药频率、检验指标单位、体重身高等单位 |

**合规专项审核（按规则库匹配）：**

| 维度 | 说明 |
|------|------|
| 适应症合规 | 药品与适应症匹配、年龄/EOS/既往治疗是否符合要求 |
| 隐私保护 | 患者个人信息是否泄露 |
| Storyline故事线 | 就诊时间、注射时间、随访时间逻辑通顺性 |
| 安全性风险 | 急性加重期用药、注射间隔合规性 |
| 科学幻灯 | 引用文献、数据一致性 |
| 信息完整性 | 随访数据、既往治疗史完整性 |

## 9. 审核报告结构

### 报告布局

```
┌─────────────────────────────────────────────────────────┐
│  📄 导出PDF 按钮（右上角固定）                            │
├─────────────────────────────────────────────────────────┤
│  报告标题：病例医学合规自动审核意见书                      │
├─────────────────────────────────────────────────────────┤
│  患者信息：编号、性别/年龄、诊断、治疗药物、适应症         │
├─────────────────────────────────────────────────────────┤
│  隐私告警（条件显示）                                     │
├─────────────────────────────────────────────────────────┤
│  统计摘要：Pass / Warning / Fail 数量                    │
├─────────────────────────────────────────────────────────┤
│  🔍 基础质控检查（蓝色背景表格）                          │
│    - 文本规范性检查                                       │
│    - 通用名规范检查                                       │
│    - 医学逻辑审核                                         │
│    - 单位规范检查                                         │
├─────────────────────────────────────────────────────────┤
│  📋 合规专项审核（普通表格）                              │
│    - 适应症合规、隐私保护、Storyline故事线                │
│    - 安全性风险、科学幻灯、信息完整性等                   │
├─────────────────────────────────────────────────────────┤
│  页脚：品牌标识 + 免责声明                                │
└─────────────────────────────────────────────────────────┘
```

### 表格列宽分配

| 列名 | 宽度 | 说明 |
|------|------|------|
| 审核维度/检查项目 | 10% | 仅显示文本 |
| 状态 | 8% | 状态徽章 |
| 依据详情/检查详情 | 42% | 详细说明 |
| 优化建议/修改建议 | 40% | 修改建议 |

### PDF 导出

- 右上角固定"导出PDF"按钮
- 点击调用浏览器 `window.print()` 打印对话框
- 打印时自动隐藏按钮（`.no-print` 样式）
- 保留表头渐变色（`print-color-adjust: exact`）
- 合并报告中每个病例自动避免分页（`page-break-inside: avoid`）

## 10. Demo 验证关注点

### 正向验证（以西甲硅油案例为例）

- 大模型应准确识别出该患者为功能性胃肠病伴腹胀、无器质性病因、无警示症状
- 对照 `rules_config.json`，应在"适应症合规"维度输出 **Pass**

### 负向注入验证（测试 Agent 纠错能力）

- 修改 PPT 中的患者年龄（如把美泊利单抗的年龄改成 10 岁）
- 或在西甲硅油病例里加入"器质性肠梗阻"
- 再次运行，检查是否触发 **Fail** 或 **Warning** 并给出修改建议

## 11. 扩展指南

### 新增药品规则

1. 编辑 `config/rules_checklist.csv`，添加新规则行
2. 同步更新 `config/rules_config.json`
3. 重新运行 `python main.py`

### 启用视觉模型

在 `.env` 中设置：
```env
OPENAI_MULTIMODAL_MODEL=your-vision-model-name
```

### 切换 API

修改 `.env` 中的 `OPENAI_BASE_URL`、`OPENAI_API_KEY`、`OPENAI_MODEL_ID` 即可，无需改动代码。
